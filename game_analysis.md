# Отчет об анализе производительности игры и оптимизации

В этом документе описано текущее состояние игры на Three.js и даны рекомендации по улучшению FPS и общей производительности.

## Текущие проблемы

### 1. Высокое количество вызовов отрисовки (Draw Calls) — Критично
В данный момент игра создает отдельный объект `THREE.Mesh` и геометрию `THREE.BoxGeometry` для каждой стены и кирпича в лабиринте.
- **Проблема**: Каждый меш приводит к отдельному вызову отрисовки на GPU. Для лабиринта 33x33 это сотни и тысячи вызовов.
- **Результат**: Огромная нагрузка на процессор, что вызывает падение FPS даже на мощных устройствах.

### 2. Слишком высокое разрешение карт теней
Разрешение теней `moonLight` установлено на `4096 x 4096`.
- **Проблема**: Это избыточно для браузерной игры. Требует много видеопамяти (VRAM) и ресурсов GPU для расчета каждого кадра.
- **Результат**: Сильное падение производительности видеокарты и возможные микрофризы.

### 3. Отсутствие повторного использования геометрии
Функция `createUnevenWall` создает новый экземпляр `THREE.BoxGeometry` для каждой стены.
- **Проблема**: Излишнее выделение памяти для идентичных данных.
- **Результат**: Высокое потребление оперативной памяти и медленная инициализация уровня.

### 4. Тени на мелких объектах
Множество мелких объектов (декоративные кирпичи, бонусы) имеют включенную опцию `castShadow = true`.
- **Проблема**: Расчет теней для множества мелких неважных объектов перегружает процесс рендеринга теней.

## Варианты решения (Сценарии реализации)

### Вариант А: Инстансинг (Instanced Rendering) — Лучший для FPS
Вместо сотен отдельных стен используется один `InstancedMesh`.

**Пример концепции кода:**
```javascript
// Создаем один инстанс-меш для всех стен
const wallGeometry = new THREE.BoxGeometry(1, 1, 1); // Базовая высота 1
const wallMaterial = new THREE.MeshPhongMaterial({ map: textures.brick });
const instancedWalls = new THREE.InstancedMesh(wallGeometry, wallMaterial, totalWallCount);

// Устанавливаем положение и МАСШТАБ для каждого экземпляра
let idx = 0;
for (let y = 0; y < mazeSize; y++) {
    for (let x = 0; x < mazeSize; x++) {
        if (grid[y][x] === 1) {
            const height = 1.1 + Math.random() * 0.5; // Разная высота
            const matrix = new THREE.Matrix4();
            
            // Настраиваем позицию и масштаб (высоту) через матрицу
            matrix.compose(
                new THREE.Vector3(x, (height / 2) - 0.5, y), // Позиция
                new THREE.Quaternion(),                      // Поворот (без поворота)
                new THREE.Vector3(1, height, 1)              // Масштаб (здесь задается высота!)
            );
            
            instancedWalls.setMatrixAt(idx++, matrix);
        }
    }
}
scene.add(instancedWalls);
```
* **Плюсы**: Всего 1 вызов отрисовки (Draw Call) для всех стен. Колоссальный прирост FPS.
* **Гибкость**: Хотя геометрия одна, через матрицу (`setMatrixAt`) можно задать **разный масштаб (высоту)** для каждой стены по отдельности.
* **Минусы**: Все стены используют один и тот же материал (текстуру). Сложнее сделать "рваные" края сверху, как сейчас (с отдельными кирпичиками), но можно имитировать это через дополнительные инстансы кирпичей.

### Вариант Б: Объединение геометрии (Geometry Merging) — Для визуального разнообразия
Объединение всех уникальных стен в одну большую геометрию перед созданием одного меша.

**Пример концепции кода:**
```javascript
import * as BufferGeometryUtils from 'three/examples/jsm/utils/BufferGeometryUtils.js';

const geometries = [];
for (...) {
   const geo = new THREE.BoxGeometry(1, randomHeight, 1);
   geo.translate(x, y, z);
   geometries.push(geo);
}
const mergedGeo = BufferGeometryUtils.mergeGeometries(geometries);
const mesh = new THREE.Mesh(mergedGeo, wallMaterial);
scene.add(mesh);
```
* **Плюсы**: 1 вызов отрисовки. Позволяет сохранять разную высоту и неровности каждой секции стены.
* **Минусы**: Требует больше памяти при генерации очень больших лабиринтов.

### Вариант В: Оптимизация теней

**Как `mapSize` влияет на тени:**
`mapSize` определяет разрешение текстуры (карты глубины), в которую "рисуются" тени с точки зрения источника света.
- **Высокий mapSize (4096):** Тени очень четкие, края не "лесенкой" даже при приближении. Но это требует **в 16 раз больше памяти**, чем 1024, и сильно нагружает видеокарту.
- **Низкий mapSize (512-1024):** Тени могут выглядеть "пиксельными" или размытыми по краям. Однако это **драматически повышает FPS**, а размытость можно сгладить программно (`THREE.PCFSoftShadowMap`).

**1. Снижение разрешения**:
Установить `mapSize` в значение `1024` или `2048`. Этого достаточно для четких теней в данной игре, и нагрузка на видеокарту упадет в разы.

**2. Адаптивное разрешение**:
```javascript
const shadowRes = isLowEndDevice ? 512 : 1024;
moonLight.shadow.mapSize.set(shadowRes, shadowRes);
```

### Вариант Г: Общий кэш ресурсов
Создание простого объекта для управления общими ресурсами.

**Пример концепции кода:**
```javascript
const ResourceCache = {
    wallGeo: new THREE.BoxGeometry(1, 1.1, 1),
    brickGeo: new THREE.BoxGeometry(0.4, 0.2, 0.25),
    getGeo(type) { return this[type]; }
};
// Использование: new THREE.Mesh(ResourceCache.getGeo('wallGeo'), material);
```
