# Анализ проблем с рендерингом (Лунный свет и Монстр)

В данном документе представлен глубокий технический анализ двух проблем, заявленных в проекте: отсутствие динамических теней от объектов между луной и стенами, а также исчезновение/невидимость монстра на уровнях.

---

## 1. Луна и ее освещение на уровнях (Shadow Clipping)

### Суть проблемы:
Ранее объекты могли создавать тени друг от друга (стены от света, объекты между стеной и светом), но эта настройка "пропала" (или перестала правильно обрабатываться). Позиция луны: `(0, 150, -250)`, в то время как цель направленного света (target) находится в центре лабиринта `(0, 0, 0)`.

Дистанция от источника света до цели рассчитывается как:
`sqrt(150^2 + 250^2) ≈ 291` единица.

### Главная причина (Root cause):
В коде игры (файл `src/main.js`, методы `updateLighting` и `updateMoonLightShadows`), камера теней (`moonLight.shadow.camera`) настроена следующим образом:
- `near = 0.1`
- `far = frustumDistance` (значение варьируется от `60` на низких до `160` на ультра-настройках).

Так как сама луна отдалена от центра лабиринта на `~291` единицу, а максимальная дальность отрисовки теней (`far`) ограничена значением `160`, то так называемый **Frustum (усеченная пирамида видимости)** камеры теней заканчивается задолго до того, как свет физически достигает стен и объектов лабиринта. Из-за этого объекты не попадают в зону расчета теней, и тени от них полностью обрезаются движком Three.js (clipping effect). 

Кроме того, размер камеры (frustumSize) от 15 до 40 — это ширина обхвата площади теней. Если лабиринт большой (например, 100x100), ширина 40 охватит только центр 80x80.

### Как пофиксить:
1. **Быстрый патч:** Увеличить `frustumDistance` (и, сооветственно, `camera.far`) как минимум до `350` или `400`. Это позволит камере покрыть дистанцию от источника света `(0, 150, -250)` до объектов к лабиринту:
   ```javascript
   this.moonLight.shadow.camera.far = 400; // или frustumDistance + 250
   ```
2. **Оптимизированное решение (Рекомендуется):** В Three.js направленный свет (DirectionalLight) для теней работает качественнее, если сам источник света "привязан" к позиции игрока (камеры). Он всегда светит сверху под нужным углом (тем же, что и у луны на небе). Таким образом зона расчета теней (`frustumSize` и `frustumDistance`) может оставаться небольшой (например, `60-100` единиц вокруг игрока). Это сохранит **высокое качество (резкость) теней** и производительность игры (так как не нужно рендерить огромную карту теней `shadowMap` на весь мир в пустоту), и при этом тени будут идеально отбрасываться любыми объектами и стенами, находящимися в поле зрения игрока.

## 2. Проблема с невидимостью монстра (Monster Visibility)

### Суть проблемы:
Монстр не появляется на уровне после загрузки. Даже несмотря на то, что объект создается, ему назначаются координаты (`0, 0.8, 1`) и устанавливается `visible = true`.

### Главная причина (Root cause):
Суть бага кроется в функции **`clearMaze()`**, которая вызывается при старте игры или генерации нового уровня. Функция должна очистить сцену от старых стен, ловушек и объектов, чтобы сгенерировать новые.

Для этого в `clearMaze()` используется метод `this.scene.traverse((object) => {...})`, который проходит по **всем узлам** сцены (включая все вложенные дочерние объекты). 
Там есть проверки, которые защищают глобальные объекты от удаления:
```javascript
object !== this.scene && 
object !== this.camera && 
object !== this.monster && // <-- сохраняем группу монстра
...
```
Исключение `object !== this.monster` позволяет оставить саму главную группу (`Group`) монстра на сцене.
**НО**, метод `scene.traverse()` идет глубже и проверяет каждый `Mesh` **внутри** монстра (например, его тело из сфер, `monsterLight`, ореол `glowSphere` и т.д.). И так как дочерние `Mesh`-объекты **НЕ** являются самой ссылкой на `this.monster` (оник просто находятся внутри нее), они не проходят эту проверку. В результате каждый дочерний кусочек монстра попадает в массив `toRemove` и в конце метода `clearMaze()` беспощадно удаляется:
```javascript
if (obj.parent) obj.parent.remove(obj);
if (obj.geometry) obj.geometry.dispose();  // Геометрия удаляется из RAM
if (obj.material) obj.material.dispose();  // Материал удаляется из RAM
```
К моменту, когда вызывается функция `spawnMonster()`, переменная `this.monster` все еще существует на сцене, но она представляет собой **абсолютно пустую невидимую группу**, так как все её визуальные меши были навсегда уничтожены кодом очистки, а их ресурсы выгружены из памяти видеокарты.

### Как пофиксить:
В функции `clearMaze()` необходимо изменить логику сбора объектов для удаления.

**Вариант решения:**
1. Можно перед добавлением в `toRemove` проверять, не является ли объект частью монстра, отслеживая его родителей.
   ```javascript
   let isProtected = false;
   let current = object;
   while(current) {
       if (current === this.monster || current === this.camera || /* и так далее */) {
           isProtected = true;
           break;
       }
       current = current.parent;
   }
   
   if (!isProtected && object !== /* protected root objects */) {
       toRemove.push(object);
   }
   ```
2. Еще проще: проходить только по `children` самой сцены (топ-левел объектам), а не с помощью рекурсивного `traverse()`. Тогда вы удалите только корневые элементы стен и лабиринта вместе со всеми их внутренностями, не затрагивая внутренности монстра (поскольку сам корневой `this.monster` будет пропущен проверкой):
   ```javascript
   // Вместо scene.traverse можно использовать простой перебор:
   const toRemove = [];
   this.scene.children.forEach((object) => {
       if (object !== this.camera && object !== this.monster && /* прочие нужные */) {
           toRemove.push(object);
       }
   });
   
   // Затем для каждого корневого объекта (типа стены) вызываем уже очистку внутренностей:
   toRemove.forEach(rootObj => {
       rootObj.traverse(child => {
           if (child.geometry) child.geometry.dispose();
           if (child.material) child.material.dispose();
       });
       this.scene.remove(rootObj);
   });
   ```
Любой из этих подходов полностью исправит баг, и монстр снова будет корректно отображаться на уровне и преследовать игрока.
